Basic Logging

The Soteria logging system provides a simple, flexible way to output diagnostic information. To get started, you need to open the Logs.Import module, which brings the L logging module into scope:

open Soteria.Logs.Import;;

The L module provides logging functions for different severity levels. The most common one is info, which logs informational messages:

L.info (fun m -> m "Starting symbolic execution");;

The logging API uses a callback pattern: you pass a function that receives a formatter function m, which you then call with a format string and arguments. This design allows the logging system to avoid formatting work when a log level is disabled.

Let's log a message with a value:

let x = 42 in
L.info (fun m -> m "The answer is %d" x);;

Log Levels

Soteria provides six log levels, from most verbose to least verbose:

  • Smt: For messages about SMT solver interactions (queries, results)
  • Trace: For very detailed execution traces
  • Debug: For debugging information
  • Info: For general informational messages
  • Warn: For warnings about potential issues
  • Error: For error conditions

Each level has a corresponding logging function in the L module:

L.trace (fun m -> m "Detailed trace: entering function");
L.debug (fun m -> m "Variable x = %d" 10);
L.info (fun m -> m "Processing completed successfully");
L.warn (fun m -> m "This operation may be slow");
L.error (fun m -> m "Failed to parse input");

The smt level is special: it's designed for logging SMT solver interactions, which can be extremely verbose but crucial for understanding solver behavior:

L.smt (fun m -> m "Querying solver: (assert (> x 0))");;

Configuration

Logging behavior is controlled by Logs.Config. You can configure:

  • The minimum log level to display
  • The output format (text to stderr or HTML)
  • Whether to always log SMT queries regardless of level
  • Whether to use colors in text output
  • Whether to hide unstable values like timestamps (useful for testing)

Here's how to configure logging programmatically:

(* Create a configuration that logs at Debug level and above *)
let log_config = Soteria.Logs.Config.make
  ~level:(Some Debug)
  ~kind:Stderr
  ~no_color:false
  ();;

(* Note: In a real application, you would call:
   Soteria.Logs.Config.set_and_lock log_config
   but we skip it here to avoid affecting the tutorial environment *)

The make function creates a configuration with these parameters:

  • ~level: The minimum level to log (None disables logging entirely)
  • ~kind: Either Stderr for text output or Html for HTML output
  • ~always_log_smt: If true, log SMT queries even when level is above Smt
  • ~no_color: If true, disable ANSI color codes in text output
  • ~hide_unstable: If true, hide timestamps and durations (for reproducible output)

For command-line applications, Soteria provides Cmdliner integration:

(* In a real application's main function *)
let () =
  let open Cmdliner in
  let config_term = Soteria.Logs.Config.cmdliner_term () in
  (* ... combine with other terms and run ... *)
  Soteria.Logs.Config.check_set_and_lock config

Organizing Logs with Sections

When logging produces a lot of output (especially in HTML mode), you can organize it into collapsible sections using with_section:

L.with_section "Initialization Phase" (fun () ->
  L.info (fun m -> m "Loading configuration");
  L.info (fun m -> m "Connecting to solver");
  L.info (fun m -> m "Initialization complete")
);;

Sections can be nested to create a hierarchical structure:

L.with_section "Analyzing Module" (fun () ->
  L.info (fun m -> m "Starting analysis");
  
  L.with_section "Function: main" (fun () ->
    L.debug (fun m -> m "Analyzing function body");
    L.debug (fun m -> m "Found 3 branches")
  );
  
  L.with_section "Function: helper" (fun () ->
    L.debug (fun m -> m "Analyzing function body");
    L.debug (fun m -> m "No branches")
  );
  
  L.info (fun m -> m "Analysis complete")
);;

When using sections in the context of symbolic execution branches, you can mark them with ~is_branch:true to help visually distinguish branch-related sections in HTML output:

L.with_section ~is_branch:true "Branch: x > 0" (fun () ->
  L.trace (fun m -> m "Assuming x > 0");
  L.debug (fun m -> m "Path is feasible")
);;

L.with_section ~is_branch:true "Branch: x <= 0" (fun () ->
  L.trace (fun m -> m "Assuming x <= 0");
  L.debug (fun m -> m "Path is feasible")
);;

Important: Do not use with_section inside symbolic processes (functions that return 'a Symex.t) that may branch, because the section will be opened once but closed multiple times. Use sections around calls to Symex.run instead.

Pretty Printing and Formatting

The Logs.Printers module provides utilities for formatting output with colors, styles, and special formatting:

Colors and Styles

You can print colored text using pp_clr:

let pp_colored ft () =
  let open Soteria.Logs.Printers in
  Fmt.pf ft "%a %a %a"
    (pp_clr `Green) "Success"
    (pp_clr `Red) "Error"
    (pp_clr `Blue) "Info";;

(* Use in logging *)
L.info (fun m -> m "%a" pp_colored ());;

Apply text styles with pp_style:

let pp_styled ft () =
  let open Soteria.Logs.Printers in
  Fmt.pf ft "%a %a"
    (pp_style `Bold) "Important"
    (pp_style `Italic) "Note";;

L.info (fun m -> m "%a" pp_styled ());;

Or combine both with pp_clr2:

let pp_fancy ft () =
  let open Soteria.Logs.Printers in
  pp_clr2 `Red `Bold ft "Critical";;

L.info (fun m -> m "%a" pp_fancy ());;

Semantic Colors

For common message types, use semantic color functions:

let pp_messages ft () =
  let open Soteria.Logs.Printers in
  Fmt.pf ft "%a, %a, %a, %a"
    pp_ok "OK"
    pp_warn "Warning"
    pp_err "Error"
    pp_fatal "Fatal";;

L.info (fun m -> m "%a" pp_messages ());;

Special Formatters

Print time durations nicely with pp_time:

let duration_seconds = 0.0123 in
let duration_seconds2 = 1.543 in

(* pp_time is a formatter function *)
L.info (fun m -> m "Quick: %a, Slow: %a"
  Soteria.Logs.Printers.pp_time duration_seconds
  Soteria.Logs.Printers.pp_time duration_seconds2);;

Show percentages with pp_percent:

let total = 100.0 in
let part = 25.0 in

(* pp_percent is a formatter function *)
L.info (fun m -> m "Completed: %a"
  Soteria.Logs.Printers.pp_percent (total, part));;

Handle singular/plural forms with pp_plural:

let pp_counts ft () =
  let open Soteria.Logs.Printers in
  Fmt.pf ft "%a, %a"
    (pp_plural ~sing:"branch" ~plur:"branches") 1
    (pp_plural ~sing:"branch" ~plur:"branches") 5;;

L.info (fun m -> m "%a" pp_counts ());;

Unstable Values

When writing tests or generating reproducible output, you can hide values that change between runs (like timestamps or durations) using pp_unstable:

let current_time = Unix.gettimeofday () in

(* If hide_unstable is true in config, prints "Completed at <timestamp>"
   instead of the actual number *)
L.info (fun m -> m "Completed at %a"
  (Soteria.Logs.Printers.pp_unstable ~name:"timestamp" Fmt.float)
  current_time);;

Logging in Symbolic Execution

Here's a complete example showing how logging works with symbolic execution:

module Symex = Symex.Make (Tiny_solver.Z3_solver);;

open Symex.Syntax;;
open Typed.Infix;;
open Typed.Syntax;;

(* A symbolic process that logs its execution *)
let analyze_value () =
  L.info (fun m -> m "Creating fresh symbolic integer");
  let* x = Symex.nondet Typed.t_int in
  
  L.debug (fun m -> m "Symbolic value: %a" Typed.ppa x);
  
  if%sat x >@ 0s then (
    L.info (fun m -> m "Branch: x > 0");
    L.debug (fun m -> m "Value is positive");
    Symex.return "positive"
  ) else (
    L.info (fun m -> m "Branch: x <= 0");
    L.debug (fun m -> m "Value is non-positive");
    Symex.return "non-positive"
  );;

(* Run the symbolic process *)
let results = Symex.run ~mode:OX (analyze_value ());;

In this example, log messages are emitted during symbolic execution. Each branch will produce its own set of log messages.

Best Practices

When to Use Each Log Level

  • Use Smt for all SMT solver interactions (queries, responses, theory lemmas)
  • Use Trace for very detailed execution flow (every step, every state transition)
  • Use Debug for information useful when debugging (intermediate values, decisions)
  • Use Info for high-level progress information (started X, completed Y)
  • Use Warn for unexpected but recoverable situations
  • Use Error for actual errors that affect correctness

Performance Considerations

Logging is designed to be efficient when disabled:

(* The lambda is only called if the log level is enabled *)
L.trace (fun m -> 
  (* This expensive computation only runs if Trace is enabled *)
  let expensive_string = String.concat ", " (List.init 1000 string_of_int) in
  m "Generated: %s" expensive_string
);;

However, be careful with work done before calling the logging function:

(* BAD: Always computes the string, even if logging is disabled *)
let expensive_string = String.concat ", " (List.init 1000 string_of_int) in
L.trace (fun m -> m "Generated: %s" expensive_string);;

(* GOOD: Only computes if needed *)
L.trace (fun m ->
  let expensive_string = String.concat ", " (List.init 1000 string_of_int) in
  m "Generated: %s" expensive_string
);;

Structured Logging Patterns

When logging complex operations, use consistent patterns:

let process_item item_name =
  L.info (fun m -> m "Processing %s" item_name);
  L.debug (fun m -> m "Loading data for %s" item_name);
  (* ... do work ... *)
  L.debug (fun m -> m "Completed %s successfully" item_name);;

process_item "configuration";;
process_item "input file";;

Use sections for major phases:

let run_analysis () =
  L.with_section "Setup" (fun () ->
    L.info (fun m -> m "Initializing solver");
    L.info (fun m -> m "Loading program")
  );
  
  L.with_section "Analysis" (fun () ->
    L.info (fun m -> m "Running symbolic execution");
    L.info (fun m -> m "Found 10 paths")
  );
  
  L.with_section "Reporting" (fun () ->
    L.info (fun m -> m "Generating report");
    L.info (fun m -> m "Writing output")
  );;

run_analysis ();;

That's it!

You now know how to effectively use Soteria's logging system! Key takeaways:

  • Use the L module from Logs.Import for all logging
  • Choose appropriate log levels for different types of information
  • Configure logging behavior with Logs.Config
  • Organize verbose output with sections
  • Use pretty printing utilities for formatted, colorful output
  • Keep performance in mind by doing expensive work inside the logging lambda

For more details, see the Logs API documentation.
