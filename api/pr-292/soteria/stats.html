Introduction to Stats

The Stats module provides a way to track statistics across your symbolic execution engine. It uses algebraic effects under the hood to maintain a statistics context without requiring you to pass a state object explicitly everywhere.

To use the statistics context, you need to wrap your execution with Stats.As_ctx.with_stats:

let (result : unit Stats.with_stats) =
  Stats.As_ctx.with_stats () (fun () ->
    (* Your execution here *)
    ()
  )

Recording Statistics

Once inside the with_stats context, you can record various types of statistics using the functions in Stats.As_ctx.

For example, you can increment counters, add floats, or measure the time taken by a function:

let (result : unit Stats.with_stats) =
  Stats.As_ctx.with_stats () (fun () ->
    (* Increment a counter named "instructions" *)
    Stats.As_ctx.incr "instructions";
    Stats.As_ctx.incr "instructions";
    
    (* Add a float value *)
    Stats.As_ctx.add_float "memory_usage" 10.5;
    
    (* Measure execution time *)
    Stats.As_ctx.add_time_of_to "heavy_computation" (fun () ->
      ()
    )
  )

After the execution finishes, you can retrieve these values from the stats field of the result using accessors like Stats.get_int and Stats.get_float:

let () =
  let instrs = Stats.get_int result.stats "instructions" in
  Format.printf "Instructions: %d\n" instrs

Customizing Output

By default, Soteria knows how to print basic statistics types. However, you can register custom printers for specific statistics using Stats.register_printer or convenience functions like Stats.register_int_printer.

let () =
  Stats.register_int_printer "instructions"
    ~name:"Instructions Executed"
    (fun _stats fmt n -> Format.fprintf fmt "%d instrs" n)

You can then pretty-print the collected statistics:

let () =
  Format.printf "%a@." Stats.pp result.stats

Conclusion

The Stats module is a powerful tool to instrument your symbolic engine. By leveraging its algebraic effects based context, you can easily track metrics like instructions executed, time spent in the solver, and more, without cluttering your main codebase.
